Use of Data and Information

Learning Objectives

  • Identify sources of data and information (internal & external) relevant to planning and risk assessment.
  • Evaluate reliability of information, including dependency on ITGCs, application controls, and key reports.
  • Apply automated tools and techniques (ATTs) and data analytics appropriately in planning and RA.
  • Address data quality, lineage, completeness, and population integrity when using data for testing or analytics.
  • Document the use of data/information and its effect on identified risks and planned responses.

1) Sources of Data & Information

  • Internal: GL, AR/AP/Inventory subledgers, fixed asset systems, revenue platforms, key reports (aging, exception, journal entry logs), EUC spreadsheets.
  • External: Bank/broker confirmations, legal letters, SOC 1® reports from service orgs, market/pricing data, regulator filings, independent actuarial/valuation reports.
  • Operational/IT: System logs, access/change records, job processing outputs that corroborate process performance.
Tip: During planning, build a simple Data Map: source → owner → extraction method → intended use → reliability checks.

2) Reliability of Information

Key Reports & EUCs

  • Identify reports used as audit evidence; validate logic (parameters, filters, completeness) or rely on tested app controls that generate them.
  • EUCs (spreadsheets): test data integrity (links, formulas, version control) and access controls.

ITGC & App Controls

  • Strong ITGCs (access, change, operations) increase confidence in automated reports.
  • Where ITGC weaknesses exist, increase substantive work or independently validate the data transformation logic.

3) Using Automated Tools & Techniques (ATTs)

  • Planning analytics: trend/ratio analyses, peer benchmarking, outlier detection to steer risk assessment.
  • Population analysis: stratify, filter, or full‑population scans (e.g., journal entry testing rules, duplicate payment checks).
  • Link to risk: Analytics inform where to test more; they don’t replace appropriate audit procedures.
  • Parameters & reproducibility: record criteria, date ranges, joins/filters, and thresholds so work can be re‑performed.
Mini‑example

Journal‑entry analytics flag unusual combinations (user at odd hours + weekend + manual JE to revenue). Response: expand JE testing and understand access controls.

4) Population Integrity & Data Quality

  • Completeness: reconcile extracted data totals to control accounts; use sequence checks (invoice numbers, JE IDs), and cut‑off windows.
  • Accuracy: spot‑check fields back to source docs; validate mappings and currency conversions.
  • Lineage: understand transformations (ETL steps); where possible, obtain read‑only extracts directly from the system of record.
  • Handle exceptions/false positives systematically; refine rules without biasing results.
Common pitfall: Using analytics outputs without first proving the dataset is complete and accurate for the stated objective.

5) External Information Sources (EIS) & SOC 1®

  • When relying on information from service organizations (payroll, revenue platforms), obtain the SOC 1® Type 2 report covering the period.
  • Evaluate complementary user‑entity controls and test that the client performed them.
  • For market/pricing data, consider provider credibility, methodology transparency, and corroborate with alternate sources where feasible.

6) Privacy, Security & Ethics

  • Protect PII and sensitive data; use secure transfer & storage; limit access by need‑to‑know.
  • Avoid creating unnecessary copies; prefer ephemeral analysis environments with audit trails.
  • Follow laws/regulations and firm policies on data retention and destruction.

7) Documentation

  • Purpose of each dataset/analytic and how it supports planning or risk assessment.
  • Source, extraction date, method, parameters; tests of completeness/accuracy; reliance on ITGCs/app controls or independent validation.
  • Findings, anomalies, and how they changed planned procedures or risk assessments.

8) Quick Checks

  1. What must you do before relying on an analytic? Validate the completeness & accuracy of the underlying dataset.
  2. How do you validate a key report? Test report logic/parameters or rely on effective app controls producing it.
  3. Do ATTs replace auditing standards? No—judgment and appropriate procedures are still required.
  4. What to do with SOC 1® user controls? Test that the client performed them; don’t assume.
